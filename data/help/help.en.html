FL Lingo Help

FL Lingo is a relocalization tool for Freelancer mods. It compares a current game install with a reference install, applies known reference text automatically, identifies mod-added content without a match, supports export and import for external translation, and writes the selected text back into the original game DLLs.

Main use case: your game with an English mod on the left, and a clean localized Freelancer installation on the right, for example a German vanilla install.

What FL Lingo does

FL Lingo is not a generic online translator. It has three core jobs:

  1. Find existing target-language text in a reference installation and apply it automatically.
  2. Identify new mod-specific text that has no target-language reference match.
  3. Write translated text back into the existing Freelancer DLLs without rebuilding them from scratch.

Recommended workflow

  1. Select Game with mod / current game.
  2. Select Reference install / comparison game.
  3. Run Compare with reference.
  4. Review the result using the purple, green, yellow, and gray progress states.
  5. Export open entries if the mod contains new text that still needs translation.
  6. Translate that file externally or use the editor for individual entries.
  7. Import the translated file.
  8. Run Apply translation.
  9. Test in game. If needed, rebuild the project and refine terminology or manual translations.

Translation logic in detail

1. Catalog building

FL Lingo reads the resource DLLs referenced by freelancer.ini in both installations, plus supplemental UI DLLs such as resources.dll. It then builds an internal resource catalog from them.

  • RT_STRING: regular string resources such as names, menu labels, and UI text.
  • RT_HTML: infocards and RDL-/HTML-like text resources.
  • Each entry stores DLL name, local resource ID, resource kind, and text.

2. Reference matching

The first match is purely technical and based on:

resource kind + DLL name + local resource ID

If the source and reference catalogs contain the same DLL, the same resource type, and the same local ID, FL Lingo treats them as a pair. This step is intentionally structural rather than heuristic.

If a mod deliberately reuses vanilla IDs for different meanings, the match can be technically correct but semantically unexpected. FL Lingo does not try to judge intent at this stage.

3. Status calculation

After matching, every entry receives a status:

  • Already in target language: source and target already match.
  • Auto transferable: there is a target text in the reference install, but the source differs.
  • Mod-only content: there is no target-side reference entry.
  • Manually translated: the entry has an imported or directly edited custom translation.

4. Terminology and patterns

For entries without a reference match, FL Lingo does not attempt unrestricted machine translation. Instead it uses a controlled combination of terminology and pattern rules.

  • Terminology: fixed term mappings such as factions, roles, ship labels, and UI terms.
  • Patterns: controlled replacements for recurring phrases and text fragments.
  • Language-specific: one terminology file is used per target language, such as terminology.de.json.

Example terminology:

Bounty Hunters Guild -> Gilde der Kopfgeldjaeger
Equipment Dealer -> Ausruestungshaendler
Jump Hole -> Sprungloch

Example patterns:

Licence -> Lizenz
Power Generator -> Energiegenerator
Welcome to the Crossfire Universe -> Willkommen im Crossfire-Universum

5. Structured entries

Many Freelancer entries are not plain sentences. FL Lingo treats several common structures separately:

  • NPC blocks: faction, role, and person name can be handled independently.
  • Label/value lines: in Armor: 10700, only the left side is translated.
  • Named locations: patterns such as the Aquarius Cloud or the Vorgha Field can be localized rule-based.

Example NPC block:

Blood Dragons
Shogun Base Equipment Dealer
Rikyu Asari

Possible automatic suggestion:

Blutdrachen
Shogun Base Ausruestungshaendler
Rikyu Asari

6. RDL / infocard logic

Long infocards are handled more conservatively than regular strings. FL Lingo no longer performs blind replacements across the full XML/RDL block. Instead it processes only the contents inside <TEXT>...</TEXT>.

  • Known labels such as DIAMETER, MASS, or CLASS are translated.
  • Known full sentences or safe phrase patterns are translated.
  • Loose single-word substitutions inside long prose are intentionally limited to avoid mixed or broken text.
This conservative behavior is intentional. A fully English paragraph is usually better than a half-translated paragraph with incorrect meaning.

7. Skip logic

Some entries should not be treated as translation work. These are counted as skipped and appear as yellow progress.

  • Pure placeholders such as #
  • Value-only lines such as 6/0, 0/10, or plain numbers
  • Obvious proper names
  • Location and station names when no actual translation is required

Status values and progress

The progress bar distinguishes multiple kinds of coverage:

  • Purple: already identical to the target reference inside the current game install
  • Green: ready to apply automatically or manually
  • Yellow: intentionally skipped
  • Gray: still open

Gray does not automatically mean “broken” or “unknown”. It only means that the entry is currently not covered by a reference match, terminology, pattern rules, or a manual translation.

Editor, terminology, and pattern management

The editor is the secondary workflow for targeted follow-up work.

  • The right text field is editable and stores manual translations in the current project.
  • Selected terms can be saved directly as terminology mappings.
  • The built-in management area allows editing terminology entries and pattern rules.
  • After such changes, use Rebuild project so the new rules are applied to the full catalog again.

Export and import for external translation

This workflow is intended for long free-form text and larger mod-specific content that should not be handled only by rule-based translation.

  1. Export open entries
  2. Translate the JSON externally
  3. Import translation
  4. Review the result in the editor
  5. Apply translation

DLL analysis

The DLL Analysis page is not just raw statistics. It evaluates the safest write strategy for each DLL and helps the user understand how FL Lingo intends to modify the install.

  • safe to replace with target DLL: the full DLL can be copied from the reference install.
  • partially relocalizable only: only matching entries will be patched.
  • not safe to replace: no automatic full replacement.

DLL writing and backups

FL Lingo no longer writes fresh resource-only DLLs. It updates the existing DLLs directly so the original resource structure, language metadata, and mod-specific content remain intact.

  • A backup is created automatically before writing.
  • The apply run can resume DLL by DLL if an error occurs.
  • The main window shows an embedded progress area while writing.
  • Restore backup... can be used to roll written DLLs back.
If you work against a live game installation, always test new translation runs on a copy first.

Limits of automation

FL Lingo is intentionally not a full machine translator. The main limits are:

  • Free-form story and lore text cannot always be translated safely with local rules.
  • Mods may deliberately remap resource IDs compared to vanilla.
  • Some already-German mod text still has no reference match and therefore remains technically open.
  • Conservative pretranslation is often better than an incorrect full automatic translation.

That is why the real core of FL Lingo is the combination of reference matching, terminology, pattern rules, and the external export/import workflow.

Languages, themes, and projects

  • UI languages: English, German, French, Spanish, Russian
  • Themes: light, dark, high contrast
  • Project files use the .FLLingo extension
  • The project stores selected paths, language pair, comparison state, and manual translations
